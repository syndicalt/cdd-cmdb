## Commands

```bash
# Install
pip install -e "."              # or: pip install httpx pytest hypothesis
pip install -e ".[generator]"   # also installs anthropic SDK for the generator

# Run the minimal suite against a live CMDB instance
CMDB_BASE_URL=http://localhost:8080 pytest -c profiles/minimal.ini

# Run the full core suite
CMDB_BASE_URL=http://localhost:8080 pytest suites/core/

# Run a single test file
CMDB_BASE_URL=http://localhost:8080 pytest suites/core/test_ci_crud.py

# Run a single test
CMDB_BASE_URL=http://localhost:8080 pytest suites/core/test_ci_crud.py::TestCICreate::test_returns_valid_uuid

# Use the thorough Hypothesis profile for release validation
CMDB_BASE_URL=http://... HYPOTHESIS_PROFILE=release pytest -c profiles/enterprise.ini

# Generate a CMDB implementation (requires ANTHROPIC_API_KEY)
python -m generator --profile minimal
python -m generator --profile standard --backend python/flask/sqlite --max-iterations 10
python -m generator --output ./my-cmdb --port 9000 --model claude-opus-4-6
```

## Architecture

This repo **is the specification**, not the software. It ships test suites that define what a valid CMDB must do. Any implementation (generated by AI, hand-written, or assembled from off-the-shelf services) that passes the tests is a compliant CMDB instance.

```
specs/              Contract artifacts (OpenAPI 3.1 + JSON Schema) — source of truth for the API surface
suites/             pytest test suites — the deliverable
  core/             CI CRUD, relationships, schema validation
  security/         Injection resistance (fuzz + classic payloads), auth enforcement
  discovery/        Bulk import, source tracking, reconciliation
  performance/      Latency SLAs and throughput benchmarks
  governance/       Runtime policy CRUD and enforcement
  audit/            CI change history (immutable audit trail)
  graph/            Multi-hop traversal, impact analysis, cycle handling
harness/            Pure Python test infrastructure: HTTP client, fixtures, Hypothesis strategies
profiles/           pytest .ini files selecting which suites to run (minimal / standard / enterprise)
generator/          CLI that drives Claude to generate a passing implementation (generate → test → fix loop)
```

**The one non-negotiable rule:** no test may import implementation code. All state setup goes through `harness/client.py` (i.e., through the API under test). This is what keeps the suite implementation-agnostic.

**`CMDB_BASE_URL`** is the only coupling between the spec and any implementation.

**`harness/client.py`** — thin `httpx` wrapper. Raises typed exceptions (`NotFoundError`, `ConflictError`, `ValidationError`) so tests assert on semantics, not status codes. Use `client.raw_post()` only in schema validation tests that need to send intentionally invalid payloads.

**`conftest.py`** (root) — `make_ci` and `make_relationship` are factory fixtures that auto-clean up after each test. `@given` tests that create state must do their own cleanup in `finally` blocks since Hypothesis runs outside the fixture teardown lifecycle.

**Hypothesis profiles** — `ci` (25 examples, fast) is the default. `release` (500 examples) is for pre-release runs. Set via `HYPOTHESIS_PROFILE=release`.

**Performance SLAs** are configurable via env vars (`CMDB_SLA_CRUD_MS`, `CMDB_SLA_LIST_MS`, etc.) with conservative defaults. Override them to match your target deployment.

**Auth tests** are skipped unless `CMDB_AUTH_ENABLED=true`. Set `CMDB_READONLY_TOKEN` to also test RBAC.

**Profiles map to compliance tiers:**
- `minimal` — core CRUD + schema validation (must-pass for any instance)
- `standard` — + discovery + audit + graph traversal
- `enterprise` — full suite including security, performance, governance

**Generator** (`python -m generator`) reads specs + tests, prompts Claude to produce a server implementation, starts it, runs the profile's tests, and iterates on failures. Requires `ANTHROPIC_API_KEY`. Output goes to `--output` dir (default `./generated/`). The generated server is a standalone Python app with its own venv — no coupling to this repo at runtime.


# Configuration Management Database (CMDB) as a Test Suite Specification
Designing a Configuration Management Database (CMDB) platform based on the principle of shipping test suites instead of fixed code or libraries would represent a radical shift toward a more dynamic, AI-driven, and adaptable system. In essence, the "CMDB platform" wouldn't be a monolithic software product but a specification artifact: a comprehensive suite of tests that define the required behaviors, interfaces, data models, and invariants. Any implementation—generated on demand by AI agents, assembled from microservices, or even sourced from existing tools—that passes these tests would be a valid "instance" of the CMDB.

This approach leverages emerging AI capabilities for code generation, cloud-native architectures, and automated verification to make CMDBs more resilient, customizable, and future-proof. Below, I'll outline what such a platform might look like, including its architecture, key components, operational flow, benefits, challenges, and some example test suite snippets.

### High-Level Architecture
- **Core Artifact: The Test Suite Repository**: The platform's "deliverable" is a versioned repository (e.g., on GitHub or a private artifact store) containing test suites written in a standardized, language-agnostic format (e.g., using tools like Cucumber for BDD-style tests, Hypothesis for property-based testing, or OpenAPI for API specs with embedded tests). These suites define the CMDB's functional and non-functional requirements without prescribing implementation details.
  
- **Dynamic Instance Generation**: Users don't install software; they use an AI orchestrator (e.g., integrated with tools like LangChain, Auto-GPT, or custom scripts) to generate or compose a CMDB instance. This could involve:
  - AI generating code for core components (e.g., database schemas, APIs).
  - Pulling in compliant off-the-shelf services (e.g., AWS DynamoDB for storage, GraphQL for querying).
  - Running the tests to validate and iterate until it passes.

- **Modular and Extensible Structure**: The platform is divided into "modules" (e.g., core CI management, discovery, reporting), each with its own test suite. Users can compose subsets for lightweight deployments.

- **Runtime Environment**: Instances run in containerized, serverless environments (e.g., Kubernetes, AWS Lambda) for easy scaling and ephemerality. If an instance fails tests (e.g., due to drift), it's regenerated automatically.

- **Governance Layer**: A meta-test suite ensures that generated instances adhere to security, compliance, and performance baselines (e.g., CIS benchmarks, GDPR data handling).

### Key Components
1. **Test Suites as the Specification**:
   - These are the heart of the platform. They'd cover:
     - **Data Model**: Tests for Configuration Items (CIs) like servers, apps, networks—ensuring attributes, relationships (e.g., "depends on"), and schemas are handled correctly.
     - **APIs/Interfaces**: REST/GraphQL endpoints for CRUD operations, queries, and integrations.
     - **Discovery and Federation**: Tests for auto-discovery (e.g., scanning cloud resources) and federating data from external sources.
     - **Analytics and Reporting**: Invariants for impact analysis, dependency mapping, and visualizations.
     - **Security and Reliability**: Fuzz tests for injections, performance under load, and failover scenarios.
   - Suites would mix example-based tests (specific scenarios), property-based tests (general rules), and oracles (self-verifying round-trips).

2. **AI-Driven Generator**:
   - A tool (itself defined by tests!) that takes the suites as input and outputs code/services. For example:
     - Prompt an LLM with: "Generate a Python/Flask app that passes these CMDB API tests, using PostgreSQL for storage."
     - Use optimization loops: Generate → Test → Fix failures → Regenerate.

3. **Verification Harness**:
   - A runner (e.g., based on pytest, Jest, or a custom Dockerized tester) that executes suites against generated instances. It could integrate with CI/CD pipelines for continuous validation.

4. **User Interface/Dashboard**:
   - Not fixed code, but tests defining UI behaviors (e.g., "Search for CI by attribute returns results in <2s"). AI generates a frontend (e.g., React app) on demand, perhaps using no-code tools like Retool as a base.

5. **Integration Ecosystem**:
   - Test suites for plugins (e.g., with ITSM tools like Jira). Users generate adapters that pass integration tests.

### Operational Flow
1. **Setup**: User clones the test suite repo and configures parameters (e.g., scale: small team vs. enterprise; backend: SQL vs. NoSQL).
2. **Generation**: Run the AI generator: "Spin up a CMDB instance compliant with v1.0 suites, optimized for AWS."
3. **Validation**: Auto-run tests; if it fails, debug via logs or regenerate.
4. **Usage**: Interact via generated APIs/UI. Data persists in a backend that passes durability tests.
5. **Update/Maintenance**: To "upgrade," pull new test suites and regenerate. No migrations needed—new instances inherently comply.
6. **Teardown/Regeneration**: Instances are disposable; regenerate for new environments (e.g., dev vs. prod).

### Benefits
- **Adaptability**: Easily tailor to environments (e.g., on-prem, multi-cloud) without forking code.
- **Reduced Vendor Lock-in**: Switch backends or generators without rewriting everything.
- **Security**: No static code means fewer supply-chain vulnerabilities; tests enforce zero-trust principles.
- **Cost Efficiency**: Pay only for runtime resources; no maintaining deprecated codebases.
- **Innovation**: Encourages experimentation—e.g., use vector DBs for semantic CI searches if tests allow.
- **Compliance**: Tests can embed regulatory checks (e.g., "Anonymize PII in queries").

### Challenges and Mitigations
- **Test Completeness**: Incomplete suites lead to flawed instances. Mitigation: Use community contributions and AI-augmented test generation (e.g., from natural language requirements).
- **Performance Overhead**: Generation takes time. Mitigation: Cache validated templates or use pre-generated "seeds."
- **Complexity for Users**: Not everyone has AI expertise. Mitigation: Provide a managed service wrapper (e.g., a SaaS that handles generation).
- **Data Migration**: Moving data between instances. Mitigation: Tests for export/import formats ensure portability.
- **Debugging**: Harder to trace issues in generated code. Mitigation: Include logging and observability tests.

### Example Test Suite Snippets for CMDB Features
Here's a taste of what the suites might look like, in Python/pytest style for clarity. These could be adapted to other frameworks.

#### 1. Core CI Management (Property-Based)
```python
from hypothesis import given, strategies as st
import json

# Strategy for generating valid CIs
ci_strategy = st.dictionaries(
    keys=st.text(min_size=1), 
    values=st.one_of(st.integers(), st.text(), st.booleans(), st.none())
)

@given(ci_strategy)
def test_ci_crud_round_trip(ci_data):
    # Oracle: create → read → update → read → delete → not found
    ci_id = create_ci(json.dumps(ci_data))  # Returns unique ID
    assert get_ci(ci_id) == ci_data
    updated = {**ci_data, "status": "updated"}
    update_ci(ci_id, json.dumps(updated))
    assert get_ci(ci_id) == updated
    delete_ci(ci_id)
    with pytest.raises(NotFoundError):
        get_ci(ci_id)

@given(st.lists(ci_strategy, min_size=10))
def test_bulk_create_performance(cis):
    start_time = time.time()
    ids = bulk_create_cis([json.dumps(ci) for ci in cis])
    assert len(ids) == len(cis)
    assert time.time() - start_time < 1.0  # Scale-appropriate perf
```

#### 2. Relationship Management (Example-Based + Invariants)
```python
def test_create_relationship():
    server_id = create_ci('{"type": "server", "name": "web1"}')
    app_id = create_ci('{"type": "app", "name": "myapp"}')
    rel_id = create_relationship(server_id, app_id, "hosts")
    assert get_relationships(server_id) == [{"target": app_id, "type": "hosts"}]

@given(ci_strategy, ci_strategy, st.text())
def test_relationship_integrity(ci1, ci2, rel_type):
    id1 = create_ci(json.dumps(ci1))
    id2 = create_ci(json.dumps(ci2))
    create_relationship(id1, id2, rel_type)
    delete_ci(id1)  # Cascade or error? Suite decides
    with pytest.raises(IntegrityError):  # Enforce no dangling refs
        get_relationships(id1)
```

#### 3. Discovery API (Security-Focused)
```python
def test_discover_cloud_resources():
    # Mock or real integration; test expects certain outputs
    results = discover("aws:ec2", {"region": "us-east-1", "creds": "valid"})  # Creds redacted in tests
    assert len(results) > 0
    assert all("instance_id" in ci for ci in results)

def test_discover_rejects_invalid_creds():
    with pytest.raises(AuthError):
        discover("aws:ec2", {"creds": "invalid"})

# Fuzz for injections
@given(st.text())
def test_discover_input_sanitization(malicious_input):
    try:
        discover("generic", {"query": malicious_input})
    except ValueError:
        pass  # No crashes or executions of malicious code
```
